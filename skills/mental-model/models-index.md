# Mental Models Index

The 32 models the skill draws from. Each row: name, core question it raises, and which problem domains it fits best.

When applying a model, USE the core question — don't paraphrase the model's definition.

---

## 🧠 Thinking Better

| Model | Core Question | Best for |
|---|---|---|
| **Inversion** | "How would you GUARANTEE failure here?" Then avoid those things. | universal · debugging · architecture |
| **Occam's Razor** | "What's the simplest explanation that fits the facts?" | debugging · diagnosis |
| **Hanlon's Razor** | "Could this be incompetence rather than malice?" | team conflicts · interpersonal |
| **Chesterton's Fence** | "Why does this exist? Don't remove until you know why." | refactoring · process change |
| **First Principles** | "Strip to atoms. What's TRUE here, not just convention?" | architecture · strategy · stuck problems |
| **Second-Order Thinking** | "Then what? And then what after that?" | decisions · architecture · policy |
| **Reversibility** | "Is this a one-way door or a two-way door?" | decisions · architecture · hiring |

## 🪤 Mind Traps

| Model | Core Question | Best for |
|---|---|---|
| **Confirmation Bias** | "What evidence would prove me WRONG? Have I looked for it?" | strategy · code review · planning |
| **Survivorship Bias** | "What about the failures we never see?" | learning from examples · strategy |
| **Sunk Cost Fallacy** | "If you didn't already invest, would you start this today?" | decisions · roadmap · relationships |
| **Anchoring** | "Is the first number I heard distorting my estimate?" | negotiation · estimation |
| **Availability Heuristic** | "Am I treating 'easy to recall' as 'common'?" | risk assessment · prioritization |
| **Dunning-Kruger** | "Am I confident because I'm skilled or because I haven't seen the depth?" | learning · estimation |

## ⏰ Time & Effort

| Model | Core Question | Best for |
|---|---|---|
| **Opportunity Cost** | "What am I saying NO to by doing this?" | prioritization · hiring · spending |
| **Pareto Principle (80/20)** | "Which 20% of inputs cause 80% of results?" | universal · optimization · debugging |
| **Parkinson's Law** | "What deadline am I giving this? What if I halved it?" | productivity · estimation |
| **Compounding** | "What tiny consistent action would matter in 5 years?" | habits · investments · skill-building |

| Model | Core Question | Best for |
|---|---|---|

| **Antifragility** | "Does this system get STRONGER from stress, or break?" | architecture · personal growth |

## Selection heuristic

When picking 5-7 for a problem:

1. **Always consider Inversion** — it's the most universal model and reveals failure modes for free.
2. **Always consider First Principles** for architecture/strategy problems where conventions might be load-bearing.
3. **Always consider Pareto** when the problem is "not enough X" — there's usually 20% doing 80% of the work.
4. **Pick 2-3 from the dominant domain** (architecture, debugging, team, etc.).
5. **Pick 1-2 surprising ones** — if the user gets the same models they'd already think of, the skill adds nothing.
